# Bard Box Standard 01 — Reading Format

## Top-Level Reading Object

Every driver must return a reading in this structure:

```json
{
  "uid": "bb-0001",
  "timestamp": "2026-04-03T14:32:10Z",
  "status": "ok",
  "data": {},
  "extended": {},
  "raw": null
}
```

---

## Required Top-Level Fields

- `uid` — unique device identifier
- `timestamp` — ISO 8601 UTC timestamp, generated by the driver; the Pi may assign one if missing
- `status` — one of: `ok`, `stale`, `error`
- `data` — normalized sensor values only
- `extended` — optional extra structured fields (always present, may be `{}`)
- `raw` — optional raw payload or `null`

No additional top-level fields are allowed beyond these six.

---

## Rules for `data`

`data` contains only canonical Bard Box channel names.

Do not include vendor-style names, UI aliases, or transport artifacts in `data`.

Do not omit channels declared in `get_capabilities()` for that device. Every declared channel must appear in `data`, using `null` if temporarily unavailable.

---

## Channel Naming Rules

| Category | Channel Names |
|----------|--------------|
| Temperature | `temp_c` |
| Relative humidity | `rh_pct` |
| Pressure | `press_pa` |
| Particulate mass (standard) | `pm1_std`, `pm25_std`, `pm10_std` |
| Particulate mass (environmental) | `pm1_env`, `pm25_env`, `pm10_env` |
| Particle counts per ft³ | `c03`, `c05`, `c10`, `c25`, `c50`, `c100` |

Rules:
- All `cXX` channels are in particles per cubic foot (count/ft³)
- No vendor-style names in normalized output (e.g. do not use `count_0p3`)
- No UI-style aliases (e.g. do not use `count_0p3um`)
- See `channel-names.md` for the full authoritative list

---

## Value Rules

- Numeric if known
- `null` if the channel is declared in `get_capabilities()` but temporarily unavailable
- Do not omit channels declared in `get_capabilities()` for that device

Example:

```json
{
  "data": {
    "temp_c": 22.4,
    "rh_pct": 41.2,
    "press_pa": 100923,
    "c03": 84140,
    "c05": 8680
  }
}
```

---

## Status Rules

- `ok` — reading is valid and current
- `stale` — last known reading, but too old to be considered fresh
- `error` — device or parse failure; reading is unusable

When `status` is `error`, all values in `data` should be `null`.

---

## Acquisition Modes

Bard Box supports both:

- session/stream-oriented instruments
- slow polled sensors

Streaming devices normally produce repeated readings after `START` until
`STOP`. Polled devices normally produce one fresh reading in response to `READ`
and do not enter continuous streaming mode.

Both acquisition styles produce the same normalized reading object. The reading
format does not change based on whether the source was streaming or polled.

For Bard Box serial protocol devices:

- `HEADER` is an official command
- `HEADER` returns exactly one `HDR,v1,...` line
- `HEADER` may be called at any time
- `HEADER` does not start streaming and does not change device state
- Header fields must match subsequent `DAT` lines exactly, in both count and order
- `READ` returns exactly one `DAT,...` line matching the current header
- `READ` may return `ERR SENSOR_FAIL` if no sample can be acquired
- `INFO`, `PING`, `STATUS`, and `HEADER` remain valid for both device classes

---

## `extended` Rules

Use `extended` for non-core but useful structured fields.

Examples:
- Sample settings
- Alarm states
- Session metadata
- Firmware-specific extras

`extended` must always be present. Use `{}` if there is nothing to report.

Example:

```json
{
  "extended": {
    "sample_time_s": 60,
    "location": 1,
    "alarm": {
      "c03": false,
      "c05": true
    }
  }
}
```

---

## `raw` Rules

- Use for the original device payload when it is useful for audit or debugging
- Otherwise `null`
- The backend and UI must not depend on `raw`
- Must be bounded — if a string, must be less than 10000 characters; must not be unbounded logs

---

## Timestamp Rule

Always UTC ISO 8601 at the driver boundary. The driver generates the timestamp;
the Pi may assign one if the driver does not provide it.

Example: `2026-04-03T14:32:10Z`

The UI may reformat timestamps for display, but drivers must always output UTC.

---

## Time Synchronization

All Bard Box systems must maintain synchronized system time via NTP.

- The Raspberry Pi is the authoritative time source
- Timestamps must be generated at the driver level on the Pi
- Devices may include timestamps, but they are not authoritative
- Target accuracy: ±100 ms or better

## Complete Example

```json
{
  "uid": "bb-0001",
  "timestamp": "2026-04-03T14:32:10Z",
  "status": "ok",
  "data": {
    "temp_c": 22.4,
    "rh_pct": 41.2,
    "press_pa": 100923,
    "c03": 84140,
    "c05": 8680
  },
  "extended": {
    "sample_time_s": 60,
    "location": 1,
    "alarm": {
      "c03": false,
      "c05": true
    }
  },
  "raw": "2026-04-03 14:32:10,00.3,00084140,00.5,00008680,+022,033,001,0060,000,*03414"
}
```

## Error Example

```json
{
  "uid": "bb-0001",
  "timestamp": "2026-04-03T14:32:10Z",
  "status": "error",
  "data": {
    "temp_c": null,
    "rh_pct": null,
    "press_pa": null,
    "c03": null,
    "c05": null
  },
  "extended": {
    "error": "serial timeout"
  },
  "raw": null
}
```

---

## Design Intent

Normalized readings use:
- Canonical channel names
- Canonical units
- UTC timestamps

The UI layer may relabel or reformat for display, but drivers must not invent local names or aliases.

---

## Implication for GoLab

GoLab should transition from exposing vendor-style field names to canonical Bard Box names.

Replace:
- `count_0p3` → `c03`
- `count_5p0` → `c50` (or `c05` depending on which channel is actually configured)

The correct channel name depends on what particle size is configured on the GT-521S — confirm before renaming.
